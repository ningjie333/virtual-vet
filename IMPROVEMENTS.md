# Virtual Vet 诊断游戏 — 多角色改进方案

> 由三位专家角色（深度游戏发烧友 / 游戏架构设计师 / 资深兽医学者）共同讨论生成
> 日期：2026-05-04
> 状态：讨论稿，不实施代码变更

---

## 一、游戏现状概要

| 维度 | 现状 |
|------|------|
| 核心玩法 | 选病例 → 看信息 → 开检查 → 看报告 → 选诊断 → 治疗 → 评分 |
| 疾病库 | 仅 2 种（肺炎、急性肾衰竭） |
| 检查系统 | 有费用消耗，检查报告含参数值/参考范围/标记 |
| 生命体征 | HR、MAP、SpO₂、RR、GFR、动脉血 pH（6项），有警告/危险阈值 |
| 时间压力 | 濒死倒计时 + 病情阶段（稳定/恶化/危重/死亡） |
| 评分 | 诊断正确有分数和评级 |
| 技术栈 | Vue 3 前端（已构建为静态文件）+ 后端 API |

---

## 二、三位专家共识：最关键的 5 个改进方向

以下按综合优先级排序，三个角色均认为重要的方向。

### 🔴 P0：疾病库扩展 + 症状重叠设计（三位一致）

**问题：** 仅 2 种疾病 = 掷硬币，没有鉴别诊断的乐趣。

**改进方向：**
- 扩展至 **15-20+ 疾病**，按系统分类（呼吸/泌尿/消化/心血管/内分泌/中毒/外伤）
- 设计**症状重叠矩阵**：不同疾病共享部分症状，但关键指标不同（如肺炎 vs 心衰都咳嗽，但一个肺部湿罗音、一个有奔马律）
- 病例数据**驱动化**（JSON Schema + API 动态加载），非硬编码，支持非开发人员创建病例
- 加入**并发症和合并症**（如糖尿病+尿路感染）
- 加入**品种特异性易感病**（杜宾易扩心、英短易HCM、德牧易胰腺萎缩）

**专家原话：**
> 游戏发烧友："疾病库扩展到15-20+并设计症状重叠矩阵——这是游戏性的根基，没有这个其他都白搭"
> 架构师："建立结构化的疾病/病例 JSON Schema，病例数据通过 API 动态加载"
> 兽医学者："每个病例设置 5-8 个鉴别诊断，玩家需要逐一排除或标记怀疑程度"

---

### 🔴 P0：引入资源约束 + 决策深度（三位一致）

**问题：** 检查费用形同虚设，玩家最优策略永远是"全开"，决策空间为零。

**改进方向：**
- 每个病例设定**初始金币预算**，检查项目差异化定价
- **分层递进检查逻辑**：基础筛查（便宜）→ 针对性检查（中价）→ 高级确诊（昂贵），跳层扣分
- 部分检查需要**等待时间**（如培养=3分钟游戏内时间），等结果期间患宠持续恶化
- 引入**"经验性治疗"选项**：不等检查结果、凭判断直接治疗——猜对高分，猜错扣分且延误

**专家原话：**
> 游戏发烧友："好游戏的核心是好取舍，当前流程中没有让你心跳加速的决策点"
> 架构师："差异化定价 + 预算约束 + 阶梯式信息释放，把'开检查→看报告→猜病'变成真正的诊断推理游戏"
> 兽医学者："检查不是随便开的，每一项都应该有明确的诊断逻辑链，训练分层递进的诊断策略"

---

### 🟠 P1：深化诊断-治疗流程（三位一致）

**问题：** 当前"选完病→治疗→结束"过于线性，缺少关键环节。

**改进方向（分两个子方向）：**

#### P1-A：诊断端——加入问诊和体格检查
- 在查看患宠信息后增加**体格检查操作界面**（视诊/触诊/听诊/体温/黏膜/CRT）
- 增加**病史采集**（发病时间、演变过程、饮食饮水、排尿排便、用药史、疫苗驱虫）
- 加入**信息噪声**：主人描述可能遗漏关键信息，需要玩家主动追问
- 引入**假设-验证系统**：玩家标记 1-3 个鉴别诊断并排列优先级

#### P1-B：治疗端——从"一个界面"到"72小时病房管理"
- 治疗阶段扩展为**72小时（游戏内时间）的动态病程管理**
- 每小时推送生命体征报告，玩家需要根据趋势**调整用药**
- 加入**治疗药物选择**（不同药物有适应症/禁忌症/副作用）、**剂量计算**（按体重mg/kg）、**给药途径**
- 加入**突发事件**（SpO₂骤降、心律失常等），考验应变能力
- **出院决策**环节：何时可以出院？需满足哪些标准？

**专家原话：**
> 游戏发烧友："治疗后的72小时才是真正的考验——并发症、药物不良反应、病情反复才见真章"
> 架构师："治疗应该有药物选择、剂量计算、给药途径，治疗错误应该产生可见后果"
> 兽医学者："真实临床中80%的诊断线索来自病史询问和体格检查，跳过这一步等于跳过了兽医最重要的思维训练"

---

### 🟠 P1：生命体征模型 + 病理生理关联（架构师 + 兽医学者强调）

**问题：** 6 项生命体征指标独立衰减，缺乏相互关联，看起来"假"。

**改进方向：**
- 构建**简化的病理生理模型**：疾病 → 指标级联变化（如肺炎 → SpO₂↓ → HR代偿性↑ → SpO₂持续低 → MAP↓）
- 用有向图或状态机描述指标间的关联
- 增加**关键缺失参数**：体温、尿量(UOP)、血糖、疼痛评分/意识状态
- 不同病例激活不同的**重点监测参数**
- 恶化曲线**参数化可配置**，同一种疾病在不同年龄/体重/品种上表现不同

**专家原话：**
> 架构师："构建病理生理模型——每种疾病定义一组参数化脚本，用有向图描述指标间的级联关系"
> 兽医学者："应该加入体温、尿量、血糖等在兽医急救中最核心的监测参数，不同病例激活不同的重点监测"

---

### 🟡 P2：评分与复盘系统（三位均提及）

**问题：** 当前评分只告诉"对/错"，没有教学价值。

**改进方向：**
- **多维度评分**：诊断准确率、检查效率、治疗及时性、治疗规范性、费用效率
- **推理路径评估**：记录玩家整个决策序列，和"最优路径"对比
- **复盘模式**：治死后进入时间线回溯，标注"你在哪一步开始走偏"
- **知识卡片收集**：每完成一个病例解锁该疾病的知识卡（流行病学/病理机制/标准方案）

**专家原话：**
> 游戏发烧友："Roguelike之所以让人die again不是因为不怕死，而是因为每次死都能学到新东西"
> 架构师："可操作的复盘报告——每局结束后显示时间线 + 关键决策点的'当时如果你做了XX会更好'"
> 兽医学者："评分系统应反映鉴别诊断过程的质量，而不只是最终答案对不对"

## 三、各专家独立建议（有分歧或补充的）

### 游戏发烧友独有建议

| 建议 | 说明 |
|------|------|
| **职业生涯模式** | 从实习生开始，用收入买设备、解锁复杂病例，实习→住院医→主治→主任晋升线 |
| **患宠参数随机化** | 同是"犬肺炎"，金毛3岁和吉娃娃12岁表现完全不同，确保每次体验不同 |
| **多人/异步社交** | 同一病例比拼得分、异步转诊合作、每周BOSS挑战 |
| **情感挑战场景** | 预后很差时安乐死是否体面？主人强烈要求治疗但医学上无意义？ |

### 架构师独有建议

| 建议 | 说明 |
|------|------|
| **状态管理重构** | 明确 Store 分层（caseStore/sessionStore/patientStore/uiStore），游戏逻辑与渲染分离 |
| **阶段机** | 用状态机管理游戏阶段流转，避免 if/else 爆炸 |
| **运行时配置** | 病例数据/评分参数/游戏常数从构建产物中抽出来，改数值不需要重新构建前端 |
| **API 版本化** | 为 `/api/v1/` 加版本前缀，新旧前端可共存 |
| **数据埋点** | 结构化记录每个决策节点的时间戳、操作类型、状态快照 |

### 兽医学者独有建议

| 建议 | 说明 |
|------|------|
| **物种特异性差异** | 不同物种的正常值范围、疾病表现模板完全不同（猫疼痛表现隐蔽、兔无呕吐反射、鸟气囊结构不同） |
| **客户沟通环节** | 诊断后需要告知主人预后、讨论治疗方案选择、获取知情同意、费用沟通博弈 |
| **病程演变叙事** | 病例应有时间线叙事（Day1-2感染进展→Day2-3临床改善→Day3-5影像滞后），治疗后安排复诊 |
| **检查-结果逻辑关联** | 检查结果应交叉验证（X线怀疑肺炎→血常规应显示中性粒细胞升高） |

## 五、一句话总结

> **核心洞察（游戏发烧友）：** 这个游戏现在是"医疗知识小测验"，要变成好玩的游戏，需要把它重塑为"在信息不完整、资源有限、时间紧迫的条件下做决策并承担后果"的策略体验。核心不是医学知识的考核，而是在不确定性中做出最优判断的智力快感。
>
> **核心洞察（架构师）：** 疾病数据驱动 + 状态管理重构是一切扩展的前提。没有这两个，后续每个新功能都会让整体复杂度指数上升。
>
> **核心洞察（兽医学者）：** 真实的兽医诊断不是"开检查→看报告→选答案"，而是"收集信息→形成假设→验证假设→做出决策→监测反馈→调整方案"的闭环。游戏目前只模拟了中间的验证环节，首尾两端恰恰是教育价值最高的部分。
